# Trace 2026-04-08 API Layer Upgrade

## 目标

把模板项目当前偏基础、且拆分略碎的 axios 封装收敛成一套更适合中小型 React 项目的请求层。

## 发现的问题

原有实现具备基础可用性，但缺少：

- 自动取消能力的明确接入点
- 更简单的重复请求处理
- 稳定的错误归一化
- 更清晰的请求选项边界
- 更直观的文件组织

## 方案权衡

候选方向主要有两类：

1. 继续保持现有实现，能力完整但拆分偏碎
2. 吸收成熟业务项目中更直观的组织方式，保留模板所需能力，去掉业务耦合

最终选择了第二种。

## 原因

- 模板项目需要具备“能直接开业务”的基础设施水平
- 但又不适合把复杂缓存、重试编排、业务中间件全部默认塞进模板
- 成熟业务项目中的请求层在可读性和职责拆分上更接近模板需要的风格

## 本轮实施

1. 保留 `ApiError`
2. 引入 `helpers/canceler.ts`，改为“重复请求取消旧请求”
3. 引入 `helpers/interceptors.ts`，拆分 token 与客户端信息注入
4. 在核心客户端中内置上传、下载方法
5. 保持请求层与 Query/AbortSignal 兼容
6. 去掉额外的 `helpers/` 目录，保持 `shared/api` 平铺
7. 更新 ADR-0003 记录新的收敛方案

## 后续收敛

在继续审视命名后，发现 `shared/api` 目录下再使用 `api-*` 前缀会造成语义重复，因此进一步收敛为：

- `client.ts`
- `request.ts`
- `error.ts`
- `query-client.ts`

保留 `helpers/` 目录，但其中只放辅助实现文件，不再重复加 `api-` 前缀。

# ADR-0005 路由守卫与懒加载

## 状态

已采纳

## 背景

模板此前已经具备基础路由结构，但仍有两个可以明显提升稳定性和模板完整度的问题：

1. 登录页自行处理“已登录用户跳转”，导致页面层承载了本应属于路由层的访问控制逻辑
2. 页面入口均为静态导入，缺少最基本的路由级代码分割

对于一个面向中型业务项目的模板来说，这两点都应该在基础工程阶段明确下来。

## 决策

本项目采用以下方案：

1. 在 `app/router/guards.tsx` 中新增 `GuestRoute`
   - 已登录用户访问登录页时，直接重定向到 dashboard
   - 登录页本身只保留表单和提交逻辑

2. 在 `app/router/router.tsx` 中为页面入口启用懒加载
   - 页面通过 `React.lazy` 按路由分片
   - 使用统一的 `RouteLoading` 作为懒加载 fallback

3. 页面文件同时保留具名导出，并增加默认导出
   - 具名导出便于本地引用和调试
   - 默认导出便于 `React.lazy` 直接接入

## 为什么这么做

### 为什么要引入 `GuestRoute`

因为“谁可以访问某条路由”属于路由层规则，而不是页面自身职责。把已登录重定向逻辑提升到 guard 后：

- 页面更薄
- 访问控制逻辑更对称
- 与 `ProtectedRoute` 形成统一心智

### 为什么模板要默认开启懒加载

因为中型业务项目的页面数量会自然增长，路由级懒加载是最基本也最稳定的代码分割方式。模板直接内置这条路径，能减少后续补救式重构。

### 为什么增加统一的 `RouteLoading`

因为每个页面各自写 loading fallback 会导致风格分散。统一路由 fallback 能让懒加载体验更稳定，也更符合模板应提供一致基础体验的目标。

## 后果

正向影响：

- 登录页职责更聚焦
- 路由守卫结构更完整
- 页面入口开始具备代码分割能力
- 模板默认产物更接近真实项目启动形态

代价：

- router 文件增加少量 `lazy` / `Suspense` 代码
- 页面文件增加默认导出约定

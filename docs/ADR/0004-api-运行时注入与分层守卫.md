# ADR-0004 API 运行时注入与分层守卫

## 状态

已采纳

## 背景

模板最初将 token 注入、语言头注入和 401 未授权处理直接写在 `shared/api` 中。这样虽然实现方便，但会带来一个明显问题：

- `shared/api` 反向依赖 `modules/auth`
- `shared/api` 反向依赖 `app/i18n`

这与当前模板的分层规则相矛盾，也会让 `shared` 从“纯基础设施层”退化成“偷偷知道业务和应用运行时状态的底层”。

同时，原有分层规则主要依赖文档和人工约束，缺少自动化校验机制。

## 决策

本项目将 API 层拆成两部分职责：

1. `shared/api`
   - 只提供纯基础设施能力
   - 负责 axios 实例、请求发送、重复请求取消、错误归一化、上传下载支持
   - 不直接读取 `app` 或 `modules` 的运行时状态

2. `app/api/interceptors.ts`
   - 在应用层注册运行时拦截器
   - 负责 token 注入
   - 负责语言头注入
   - 负责 401 未授权后的登出策略

同时，在 ESLint 中加入最小可执行的分层守卫：

- `shared` 禁止导入 `app`、`modules`、`pages`、`widgets`
- `modules` 禁止导入 `app`、`pages`、`widgets`
- `pages` 禁止导入 `app`

## 为什么这么做

### 为什么不让 `shared/api` 直接读 store 和 i18n

因为基础设施层应该提供能力，而不是直接绑定具体应用运行时。把 token 和 locale 的来源固定死在 `shared` 中，会让：

- 测试变重
- 替换认证实现更困难
- 分层规则失去约束力

### 为什么选择“应用层注入”

因为认证、语言和未授权处理本质上都属于应用运行时策略，而不是 HTTP 底盘能力。由 `app` 注入后，依赖方向会重新回到：

- `app` -> `shared`
- `app` -> `modules`

而不是：

- `shared` -> `app`
- `shared` -> `modules`

### 为什么先使用 ESLint 守卫

因为模板最需要的是“轻量且能立即执行”的规则。`no-restricted-imports` 足以覆盖当前最关键的层间违规，并且能直接纳入现有 `npm run lint` / `npm run validate` 流程。

## 后果

正向影响：

- `shared/api` 的边界更纯净
- 认证和语言策略的归属更清晰
- 分层违规能在本地和 CI 里自动报错

代价：

- 应用启动时需要额外注册一次 API setup
- 边界守卫目前是“核心层级版”，后续仍可继续细化到 widgets 等更严格的层间规则

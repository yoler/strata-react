# ADR-0001 项目结构与分层

## 状态

已采纳

## 背景

为了让人类工程师与 AI Agent 在同一个仓库内稳定协作，需要一套足够清晰、足够轻量、并且真正能落地的目录约定。

历史上常见的问题包括：

- 入口文件与应用装配逻辑混在一起
- i18n 初始化和语言资源混放
- 通用 UI 组件和业务组件放在同一个目录
- 页面层承担过多业务逻辑

## 决策

本项目采用以下最小分层：

- `src/main.tsx`：薄入口
- `src/app`：应用装配层
- `src/locales`：语言资源
- `src/modules`：业务模块
- `src/pages`：页面层
- `src/widgets`：应用级组合块
- `src/shared`：共享层

同时采用以下规则：

1. `src/main.tsx` 只挂载 `src/app/app.tsx`
2. i18n 初始化放在 `src/app/i18n`
3. 语言资源放在 `src/locales`
4. 全局通用 UI 组件统一放在 `src/shared/ui`
5. 业务组件统一放在 `src/modules/*/components`

同时采用以下命名规则：

6. 目录与普通文件统一使用 `kebab-case`
7. store 文件使用 `xxx-store.ts`
8. 类型文件使用 `xxx.types.ts`
9. schema 文件使用 `xxx.schema.ts`
10. locale 文件保留标准语言代码格式

## 后果

正向影响：

- 入口更容易找到
- 目录边界更统一
- i18n 结构更清晰
- 通用 UI 与业务组件边界更明确
- AI 更容易把代码放到正确位置

代价：

- 与部分社区默认目录不完全一致
- 需要在文档中明确说明约定

## 备注

如果未来接入新的 UI 生成工具或代码生成器，需要优先评估其目录约定是否与 `shared/ui` 冲突，再决定是否调整结构。

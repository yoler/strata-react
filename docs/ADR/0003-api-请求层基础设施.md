# ADR-0003 API 请求层基础设施

## 状态

已采纳

## 背景

模板项目原有请求层虽然已经补过一轮能力，但文件拆分偏碎，阅读路径较长，和模板“简单可读、直接上手”的目标不完全一致。

综合参考成熟业务项目中的请求层实践后，发现其中有几项做法很适合模板吸收：

- 请求拦截器职责拆分
- 重复请求处理逻辑集中管理
- 一个核心客户端文件对外暴露统一方法
- 上传、下载等常见能力直接内置

同时也要避免把业务耦合一起带进模板，例如：

- 风控验证流程
- 全局 Message 提示
- 项目专属错误码和业务跳转

因此本轮重新收敛请求层，目标是“保留成熟项目的清晰结构，但不把业务绑定进模板”。

## 决策

本项目采用“清晰优先”的请求层基础设施方案：

1. 保留 axios 作为传输基础
2. 增加统一错误归一化，输出稳定的 `ApiError`
3. 对重复请求采用“新请求取消旧请求”的简单策略
4. 显式支持 `signal`，与 `AbortController` / TanStack Query 取消机制兼容
5. 将 token 注入、客户端信息注入、重复请求取消拆成平铺的 API 基础设施文件
6. 保持业务调用面简洁，不引入过重的业务中间层

## 具体规则

1. 请求层统一通过 `shared/api/request.ts` 暴露
2. axios 实例统一通过 `shared/api/client.ts` 管理
3. 业务代码不直接散写 axios 请求
4. 默认启用重复请求取消；可用 `skipCancel` 显式跳过
5. 可用 `skipAuth` 跳过自动鉴权头注入
6. 鉴权、取消、错误归一化、上传、下载在基础设施层处理

## 为什么这样做

### 为什么要统一错误归一化

因为业务层更关心“能不能稳定拿到 `message/status/isCanceled`”，而不是区分 axios 各种原始错误形态。

### 为什么要支持取消

Axios 官方支持 `AbortController`，TanStack Query 也会把 `AbortSignal` 传给 query function。把这条链路打通后，请求层更容易和 Query 协同工作。  
参考：

- [Axios Cancellation](https://axios-http.com/docs/cancellation)
- [TanStack Query Cancellation](https://tanstack.com/query/latest/docs/framework/react/guides/query-cancellation)

### 为什么重复请求处理改成“取消旧请求”

因为这种策略更直观，也更接近很多业务项目的常见实现：

- 相同请求再次发起时，旧请求立即失效
- 新请求保持最新语义
- 代码路径更短，更容易理解和排查

相比“共享 in-flight promise”，它在模板场景里更容易读懂。

### 为什么平铺在 `shared/api` 下，而不是再加一层 `helpers`

因为模板需要的是“职责清楚”，不是“文件越多越专业”。

对于当前体量，单独增加 `helpers/` 目录会让路径更深，但不会明显提升理解成本，因此保持 `shared/api` 平铺更统一。

因此只保留两层理解：

- 核心客户端：`client.ts`
- 对外入口与配套基础设施：`request.ts`、`error.ts`
- helper：`helpers/canceler.ts`、`helpers/interceptors.ts`

### 为什么不使用 `api-*` 前缀命名

因为目录本身已经表达了“这里是 API 层”。

在 `shared/api/` 目录下继续使用 `api-client.ts`、`api-request.ts` 这类命名，会形成语义重复。  
更优雅的做法是让：

- 目录表达领域
- 文件表达职责

因此最终采用：

- `client.ts`
- `request.ts`
- `error.ts`
- `query-client.ts`

## 后果

正向影响：

- 业务层的请求调用更稳定
- API 层结构更接近真实项目，但更易读
- 更容易对接 Query、取消、统一错误展示
- 模板的 API 基础设施更接近“能直接上手开发”的状态

代价：

- 仍然比最简版多一层基础设施
- 需要维护少量 helper 与错误模型

# ADR-0002 状态归属与浏览器存储

## 状态

已采纳

## 背景

模板项目需要同时处理以下几类状态：

- 服务端返回的数据
- 项目自身的客户端状态
- 平台型配置，例如主题和语言
- 浏览器持久化存储

讨论过程中出现了两个候选方向：

1. 把 `locale`、`theme` 等全局配置也统一塞进 Zustand
2. 让不同类型的状态交给更合适的宿主管理，再用一层轻量存储封装统一 key 与读写入口

如果把 `locale` 和 `theme` 都再包一层 Zustand，会带来几个问题：

- `locale` 会在 `i18n` 和 Zustand 中形成双状态源
- `theme` 会在 `next-themes` 和 Zustand 中形成双状态源
- 状态同步逻辑增多，职责边界变模糊
- 模板复杂度上升，但收益有限

## 决策

本项目采用以下状态归属规则：

1. 服务端状态由 TanStack Query 管理
2. 项目自身客户端状态由 Zustand 管理
3. 平台型配置优先由宿主库管理，Zustand 不重复持有同一份状态
4. `theme` 由 `next-themes` 管理
5. `locale` 由 `i18n` 管理
6. 浏览器持久化存储通过轻量封装统一管理
7. 所有浏览器存储 key 统一收敛到 `shared/constants/storage-keys.ts`

## 实现方式

- 新增 `shared/constants/storage-keys.ts` 统一定义 key
- 新增或完善 `shared/lib/storage.ts` 作为浏览器存储封装
- `i18n` 初始化优先读取封装后的存储 key，没有命中时再回退到浏览器语言和默认语言
- `next-themes` 使用统一的 `storageKey`
- Zustand `persist` 的 `name` 也使用统一 key 常量

## 后果

正向影响：

- 避免 `locale`、`theme` 出现双状态源
- 让 Zustand 职责更聚焦
- 让存储 key 可枚举、可审计、可统一改名
- 让模板更适合作为长期复用的架构起手式

代价：

- 需要多维护一层很轻的存储封装
- 需要在文档中明确区分“宿主管理”和“项目状态管理”

## 备注

如果未来出现某个全局配置没有明确宿主，且确实需要作为项目客户端状态统一协调，可以再评估是否进入 Zustand，而不是默认直接塞进 store。
